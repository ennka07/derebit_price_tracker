# Deribit Price Collector

Сервис для сбора, хранения и предоставления данных о ценах криптовалют (BTC, ETH) с биржи Deribit. Проект реализован на стеке **FastAPI + Celery + PostgreSQL + Redis** и полностью контейнеризирован с помощью **Docker**.

## 🚀 Возможности

- **Периодический сбор данных:** Автоматический сбор индексных цен каждые 60 секунд через Celery Beat.
- **REST API:** Endpoints для получения последних цен, истории и данных за диапазон времени.
- **Надежность:** Механизм повторных попыток (retry) задач Celery при ошибках сети или API (max 3 попытки).
- **Контейнеризация:** Легкий запуск всех зависимостей (БД, брокер, воркеры, API) через `docker-compose`.
- **Валидация данных:** Использование Pydantic для валидации входных и выходных данных, ограничение тикеров (`btc_usd`, `eth_usd`).
- **Точность данных:** Использование типа `Numeric` в БД для хранения финансовых данных без потерь точности.

## 📋 Требования

## 🛠 Установка и запуск

### 1. Клонирование репозитория

```bash
    git clone <repository_url>
    cd <project_directory>
```


### 2. Настройка переменных окружения
Создайте файл .env в корне проекта на основе примера .env_example:

```bash
    cp .env_example .env
```
При необходимости отредактируйте значения (пароли, порты), хотя стандартные 
значения подходят для локального запуска через Docker Compose.


### 3. Запуск сервисов
Для запуска всех сервисов (API, Worker, Beat, DB, Redis) выполните:

```bash
    docker-compose up --build
```
> **Примечание:** Сервис init-db автоматически создаст таблицы в PostgreSQL 
> при первом запуске и завершит работу.


### 4. Проверка работоспособности

- **API Health Check:** http://localhost:8000/health
- **Swagger UI:** http://localhost:8000/docs


### 5. Остановка

```bash
    docker-compose down
```

## 📡 Использование API
Основные endpoints доступны по префиксу `/api/v1/prices`.

### Получить все цены (с лимитом)

```bash
    GET /api/v1/prices/all?ticker=btc_usd&limit=100
```

### Получить последнюю цену

```bash
    GET /api/v1/prices/latest?ticker=eth_usd
```

### Получить цены за диапазон времени

```bash
    GET /api/v1/prices/range?ticker=btc_usd&start_timestamp=1700000000&end_timestamp=1700000100
```


## 🏗 Design Decisions (Архитектурные решения)
В этом разделе описаны ключевые решения, принятые при разработке архитектуры проекта.

### 1. Слоистая архитектура (Layered Architecture)
Проект разделен на логические слои для обеспечения поддерживаемости и тестируемости:

- **`app/api`**: Слой представления (FastAPI routers, deps). Отвечает за HTTP запросы/ответы.
- **`app/domain`**: Слой бизнес-логики и схем (Pydantic models). Не зависит от инфраструктуры.
- **`app/infrastructure`**: Слой реализации (БД, Celery, внешние API). Содержит репозитории, модели SQLAlchemy и клиенты внешних сервисов.
- **`app/core`**: Конфигурация приложения (настройки через `pydantic-settings`).


### 2. Celery для фоновых задач и планирования
Для сбора данных выбран Celery вместе с Celery Beat.

- **Причина**: Сбор данных — это периодическая задача (cron), которая не должна блокировать основной API.
- **Надежность**: Настроен механизм `retry` (максимум 3 попытки с задержкой 30 секунд) на случай временных проблем с сетью или API Deribit.
- **Брокер**: Redis используется как брокер сообщений и бэкенд результатов благодаря своей скорости и простоте настройки в Docker.


### 3. База данных и ORM

- **PostgreSQL**: Выбран как надежное реляционное хранилище для временных рядов цен.
- **SQLAlchemy (ORM)**: Используется для абстракции работы с БД.
- **Типы данных**: Для цены использован тип `Numeric(20, 2)`, чтобы избежать ошибок округления, 
характерных для типов с плавающей точкой (`float`), что критично для финансовых данных. 
Для времени использован `BigInteger` (UNIX timestamp).
- **Sync vs Async**: В текущей реализации API и Celery задачи используют синхронную сессию (`SyncSessionLocal`). 
Это упрощает код репозиториев и задач Celery, так как Celery воркеры по умолчанию синхронные. Асинхронный движок используется для миграции БД в init-db.


### 4. Контейнеризация (Docker Compose)
Все сервисы изолированы в контейнерах:

- **`api`**: FastAPI приложение (Uvicorn).
- **`worker`**: Celery worker для выполнения задач сбора.
- **`beat`**: Celery beat для планирования задач.
- **`init-db`**: Одноразовый контейнер для миграции БД (создание таблиц) перед стартом основных сервисов.
- **`postgres & redis`**: Инфраструктурные зависимости.

Такой подход гарантирует, что окружение разработчика идентично продакшену, и упрощает развертывание.


### 5. Внешнее API (Deribit)
Используется публичный метод `public/get_index_price`.

- **Валидация**: На уровне кода (`index_price.py`) проверяется допустимость тикеров (`btc_usd`, `eth_usd`), чтобы избежать лишних запросов к внешнему сервису.
- **Таймауты**: Установлен жесткий таймаут запроса (10 секунд), чтобы задача Celery не зависала бесконечно при проблемах на стороне биржи.

    ├── .env_example                 # Шаблон переменных окружения

## 🔧 Переменные окружения

| Переменная | Описание | Значение по умолчанию | Обязательно |
|------------|----------|-----------------------|-------------|
| `CELERY_BROKER_URL` | URL подключения к Redis брокеру | `redis://localhost:6379/0` | Да |
| `CELERY_RESULT_BACKEND` | URL бэкенда результатов Celery | `redis://localhost:6379/1` | Да |
| `POSTGRES_USER` | Пользователь базы данных | `user` | Да |
| `POSTGRES_PASSWORD` | Пароль пользователя БД | `password` | Да |
| `POSTGRES_DB` | Имя базы данных | `deribit_prices` | Да |
| `POSTGRES_HOST` | Хост базы данных (для Docker) | `postgres` | Нет |
| `POSTGRES_PORT` | Порт базы данных | `5432` | Нет |
